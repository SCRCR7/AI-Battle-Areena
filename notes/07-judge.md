# 08 - Judge Node (AI Battle Arena)

# What is the Judge Node?

The Judge Node is responsible for **evaluating** the responses generated by the two AI models.

It does **not** generate new solutions.

Instead, it compares the existing solutions, scores them, and decides which one is better.

Think of it as a teacher grading two students' answers.

---

# Why do we need a Judge?

Imagine our project without a judge.

```
User

↓

Mistral

↓

Answer


User

↓

Cohere

↓

Answer
```

Now the user has two answers.

Which one is better?

Our application doesn't know.

So we add another AI whose only responsibility is to compare both answers.

```
Question

↓

Mistral

↓

Answer


↓

Cohere

↓

Answer


↓

Judge

↓

Winner
```

---

# Mental Model

Imagine two students taking an exam.

```
Student 1

↓

Answer


Student 2

↓

Answer


↓

Teacher

↓

Scores

↓

Winner
```

The teacher doesn't answer the question.

The teacher only evaluates the answers.

Our Judge Node works exactly like the teacher.

---

# Responsibility of the Judge Node

The Judge Node should only do these things:

- Read the original question.
- Read Solution 1.
- Read Solution 2.
- Send everything to the Judge Agent.
- Receive structured scores.
- Store the result in the State.

Notice something important...

The Judge Node **does not generate solutions**.

That is the Solution Node's job.

---

# What does the Judge Node receive?

By the time the Judge Node runs,

the Solution Node has already updated the State.

Example State

```ts
{
    messages:[...],

    solution_1:"React is a JavaScript library...",

    solution_2:"React is an open-source UI library...",

    judge_recommendation:{...}
}
```

So the Judge Node already has everything it needs.

---

# Reading the State

Inside the Judge Node

```ts
const {

    solution_1,

    solution_2

} = state;
```

We can also read the original question

```ts
const question =

state.messages.at(-1)?.content;
```

Now we have

- Original Question
- Solution 1
- Solution 2

Everything required for judging.

---

# Why don't we ask Mistral or Cohere to judge?

Because each AI model would probably prefer its own answer.

Imagine asking Student 1

to grade Student 1.

Not fair.

Instead,

we use a third AI model.

```
Mistral

↓

Answer


Cohere

↓

Answer


Gemini

↓

Judge
```

This makes the evaluation more independent.

---

# Creating the Judge Agent

Example

```ts
const judge = createAgent({

    model:geminiModel,

    responseFormat:...,

    systemPrompt:...

});
```

Notice

This is exactly the same Agent we learned in the previous chapter.

The only difference is its job.

Instead of solving,

it evaluates.

---

# The System Prompt

Our System Prompt tells Gemini

how it should behave.

Example

```text
You are an AI Judge.

Compare both solutions.

Score each solution from 0 to 10.

Explain your reasoning.

Choose the better solution.

Return structured data.
```

This makes Gemini behave like an examiner instead of a chatbot.

---

# Sending the Data

We invoke the Agent

```ts
await judge.invoke({

    messages:[

        new HumanMessage(...)

    ]

});
```

The Human Message contains

```text
Question:

What is React?

Solution 1:

...

Solution 2:

...
```

Gemini now has all the information required to compare them.

---

# What does the Judge return?

Because we used

```ts
responseFormat
```

the response is structured.

Example

```json
{
    "solution_1_score":8,

    "solution_2_score":9,

    "recommended_solution":{

        "winner":"solution_2",

        "reason":"Solution 2 was more complete."
    }
}
```

This is much easier for our application to use than free-form text.

---

# Updating the State

After receiving the response

the Judge Node returns

```ts
return{

    judge_recommendation:

    judgeResponse.structuredResponse

}
```

LangGraph automatically merges it into the State.

Final State

```ts
{

    messages:[...],

    solution_1:"...",

    solution_2:"...",

    judge_recommendation:{

        ...

    }

}
```

# How does the Judge score?

The scores are **not calculated using math**.

Gemini reads both answers and evaluates them using reasoning.

Typical criteria include:

- Correctness
- Completeness
- Clarity
- Accuracy
- Relevance
- Following the user's request

Example

Question

```
Explain React.
```

Solution 1

```
React is a library.
```

Solution 2

```
React is an open-source JavaScript library developed by Meta for building user interfaces using reusable components.
```

Possible Result

```
Solution 1

7/10

Too short.


Solution 2

10/10

More complete and accurate.
```

The scores come from Gemini's evaluation, guided by your system prompt.

# Why do we store the recommendation?

Suppose the frontend wants to display

```
Winner

↓

Solution 2
```

or

```
Solution 1

8/10

Solution 2

9/10
```

Instead of asking Gemini again,

we simply read

```ts
state.judge_recommendation
```

The result is already stored.

# Common Mistakes

## Letting the Solution Node judge

Wrong.

Each node should have one responsibility.

Solution Node → Generate answers

Judge Node → Evaluate answers

---

## Forgetting to include the original question

The Judge needs context.

Always send

- Question
- Solution 1
- Solution 2

---

## Using free-form responses

Avoid

```
Solution 2 seems better.
```

Prefer structured output

```json
{
    "winner":"solution_2"
}
```

---

## Thinking the Judge modifies previous solutions

It doesn't.

The Judge only reads them.

It never changes them.

---

# Interview Questions

## Why do we need a Judge Node?

To compare multiple AI-generated solutions and choose the best one.

---

## Why use a different AI model as the Judge?

Using a separate model provides a more independent evaluation and avoids having a model assess its own answer.

---

## What information does the Judge receive?

The original question, Solution 1, and Solution 2.

---

## Does the Judge generate new solutions?

No.

It only evaluates existing solutions.

---

## Where is the Judge's result stored?

Inside the LangGraph State under `judge_recommendation`.

---

# Summary

The Judge Node is the evaluator of our AI Battle Arena.

It doesn't solve the user's problem.

Instead, it reads the solutions generated by other models, compares them, scores them, explains the reasoning, selects a winner, and stores the result in the shared State.

By separating generation and evaluation into different nodes, our graph stays clean, modular, and easier to maintain.

---

# Things I Should Remember

- The Judge Node **evaluates**, it does not generate.
- It reads data from the shared State.
- It needs the original question and both solutions.
- A third AI model (Gemini) acts as the judge.
- The System Prompt defines the judging rules.
- `responseFormat` ensures structured results.
- The Judge returns `judge_recommendation`.
- LangGraph merges the recommendation into the State.
- The frontend can directly read the stored recommendation.
- Think of the Judge Node as a **teacher grading exam papers**, not a student answering the exam.
